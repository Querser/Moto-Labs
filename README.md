# Moto Laps

> Версия 0.9.0: загруженная гонка обрабатывается в два прохода. MOG2 и YOLOX
> примерно на 3 FPS быстро находят окна с мотоциклами; внутри них YOLOX/ByteTrack
> сохраняют исходные 30 FPS, а пустые части файла полностью пропускаются. Короткие однозначные
> разрывы трека объединяются, а частичные чтения соседних кадров могут дать полный
> номер. PP-OCRv6 medium и PP-OCRv5 mobile работают локально. Нестабильный
> авторегрессионный verifier исключён из production-пути. В прямом эфире
> GoPro/USB-камера, CV и запись работают раздельно через ограниченные очереди.

Живой поток намеренно не замедляется при появлении мотоцикла. Замедление
предпросмотра не создаёт дополнительных исходных кадров и могло бы скрыть
следующий проезд. Вместо этого система сохраняет короткий буфер лучших резких
кандидатов щитка в памяти, повторно проверяет их при пересечении и затем удаляет.
Время круга всегда берётся из метки исходного кадра, а не из момента завершения
OCR.

Локальная система фиксации кругов мотоциклов по видео. Приложение обнаруживает
мотоцикл, сопровождает его между кадрами, распознаёт стартовый номер только внутри
передней части мотоцикла и создаёт запись круга в момент пересечения виртуальной
финишной линии. Интернет и облачные API во время гонки не используются.

## Быстрый запуск в Windows

Дважды щёлкните `Start Moto Laps.bat`. Лаунчер:

1. создаст `.venv`, если его ещё нет;
2. установит зависимости и применит миграции SQLite;
3. запустит сервер только на `127.0.0.1:8000`;
4. откроет веб-интерфейс в браузере по умолчанию.

Повторный двойной щелчок не создаёт второй процесс, а открывает уже запущенное
приложение. Красная кнопка «Завершить работу» в интерфейсе корректно останавливает
сервер и освобождает камеру.

Ручной запуск из PowerShell:

```powershell
powershell.exe -NoProfile -ExecutionPolicy Bypass -File .\run.ps1
```

Фоновый запуск с открытием браузера:

```powershell
powershell.exe -NoProfile -ExecutionPolicy Bypass -File .\run.ps1 -Background -OpenBrowser
```

## Настройка перед гонкой

1. Подключите USB-камеру до запуска приложения. Для GoPro сначала запустите
   GoPro Webcam, включите GoPro, выберите на камере USB Connection → GoPro
   Connect (если такой пункт есть у модели), затем подключите её кабелем USB-C.
2. В разделе «Камера» выберите устройство и сохраните его. Для проверки без
   камеры выберите «Демо-анимация».
   В Windows список строится по реальным DirectShow-именам и индексам, поэтому
   подписи больше не сопоставляются с камерами по ненадёжному порядку PnP.
   GoPro отображается отдельным пунктом «GoPro Webcam (виртуальная камера)».
   Если пункт есть, но изображения нет, проверьте, что значок GoPro Webcam в
   системном трее показывает активную камеру, а Zoom/OBS/браузер её не заняли.
3. Камеру рекомендуется закрепить перед траекторией на высоте около 2 м и
   наклонить вниз на 45–60°. Мотоцикл должен приближаться к камере, а не быстро
   проходить поперёк кадра.
4. Перетащите любой конец финишной линии отдельно либо возьмитесь за любой участок
   самого отрезка и переместите линию целиком. Изменение автоматически сохраняется
   после отпускания мыши, пера или пальца.
   Линия хранится в нормализованных координатах, поэтому изменение разрешения
   превью её не сдвигает.
5. Номерная табличка должна быть спереди, контрастной, крупной, без бликов и
   читаемой хотя бы на нескольких кадрах до пересечения.
6. Создайте гонку, укажите название и количество кругов, затем нажмите «Старт».

Для обработки записи откройте раздел «Видео», загрузите MP4/MOV/AVI/MKV, проверьте
линию, затем в настройке гонки выберите источник «Загруженное видео». Файл хранится
под случайным именем в `data/uploads/`; пустые, повреждённые, слишком большие файлы
и неподдерживаемые расширения отклоняются. Все кадры быстро декодируются в первом
проходе, но YOLO запускается примерно на 3 FPS и только при движении либо во время
периодической полной проверки. Найденные детекции объединяются в окна с запасом
до и после появления. Второй проход анализирует каждый исходный кадр только в
этих окнах. Время кругов берётся из позиции кадра в исходном видео, а не из
скорости работы компьютера. Кнопка «С начала» повторно запускает тот же файл;
идемпотентность SQLite не даёт повторно записать уже обработанное пересечение.

Портретная запись помещается в окно целиком без обрезки. Таймер между серверными
отметками интерполируется с фактической скоростью обработки исходных кадров,
поэтому минуты, секунды и миллисекунды изменяются плавно.

Встроенный видеоплеер воспроизводит файл непрерывно и синхронизируется с позицией
фонового анализа. Поэтому тяжёлый CPU-инференс больше не показывается как череда
стоп-кадров. При этом анализ по-прежнему сохраняет временную метку каждого исходного кадра и его
временную метку. В прямом эфире камера также не замедляется: захват и MJPEG-превью
работают отдельно от bounded-очереди инференса; при отставании удаляются только
устаревшие кадры обработки, а не кадры живого изображения.

Запись круга создаётся по времени реального пересечения, интерполированному между
двумя соседними кадрами. Время, затраченное на распознавание после пересечения, к
результату не прибавляется. Итог участника равен сумме всех его кругов; пауза гонки
из расчёта исключается.

## Пайплайн

```text
камера или загруженное видео -> Frame с исходной временной меткой -> MOG2-кандидат
       -> YOLOX-Tiny motorcycle -> ByteTrack -> передняя область/номерная табличка
       -> frame selector -> RapidOCR + PP-OCRv5 text detector + PP-OCRv6 medium digits
       -> многокадровый консенсус и сборка частичных чтений между треклетами
       -> ведущая грань bbox -> интерполированное пересечение линии
       -> защита от дублей -> сервис кругов -> SQLite -> WebSocket/UI
```

MOG2 не решает, является ли объект мотоциклом: он только позволяет пропускать
часть дорогих запусков нейросети, когда кадр статичен. Семантическое решение всегда
принимает YOLOX. Детектор периодически запускается и без сигнала движения, поэтому
остановившийся объект не теряется. ByteTrack переживает краткие пропуски детектора.

OCR выполняется только по перспективно выправленным кандидатам передней номерной
таблички, светлым областям с сохранённым контекстом и четырём перекрывающимся верхним фронтальным кропам внутри
bbox мотоцикла. Текст не обязан находиться в центре кропа: пересечение определяется
геометрией мотоцикла, а не положением цифр. На обычном кадре используется быстрый
recognition-only путь. Тяжёлый текстовый детектор запускается только по ограниченному
набору лучших кадров после пересечения или ухода трека; 0,3 секунды кадров после
линии уточняют номер, не меняя уже зафиксированное время круга. Маленький кроп увеличивается до OCR без изменения
геометрии исходного кадра. Если быстрый PP-OCRv4 не даёт устойчивого ответа,
официальный PP-OCRv5 mobile text detector локализует строку в лучших разновременных
кропах, а PP-OCRv6 medium распознаёт только чистую строку цифр. Короткие
пространственно однозначные разрывы ByteTrack объединяются, поэтому части одного
номера с соседних кадров могут дать полную строку; неоднозначные параллельные
мотоциклы не склеиваются. Авторегрессионная VLM-модель не входит в production-путь.
Текст на задней части бокового проезда отсекается по траектории.
Номер сохраняется строкой, поэтому `007` и `7` —
разные участники. Одиночное противоречивое чтение не создаёт круг.

## Модели и данные

В `models/yolox_tiny.onnx` находится официальный предобученный YOLOX-Tiny для
COCO; используется только класс `motorcycle` (COCO id 3). RapidOCR поставляет
быстрый PP-OCRv4/PP-OCRv5 путь, PaddleOCR 3.7.0 — PP-OCRv6 medium recognition и
PP-OCRv5 mobile detection. Florence-2 был оценён как экспериментальный verifier,
но отключён после контрольного прогона из-за отсутствия прироста и неограниченной
задержки worker-процесса. Полное происхождение, версии,
лицензии, контрольные суммы и локальные изменения перечислены в
`docs/model-provenance.md`. Тестовый снимок не используется для обучения.

Собственное обучение не выполнялось: для базового детектора имеется официальный
вес, который успешно обнаруживает мотоцикл на независимом тестовом изображении.
Для производственного качества именно на выбранной камере следует собрать
размеченное видео с реальными номерами, углами 45–60°, скоростями и освещением, а
затем отдельно дообучить детектор номерных табличек/распознаватель.

## Проверки разработчика

```powershell
.\.venv\Scripts\python.exe -m pytest -q
.\.venv\Scripts\python.exe -m ruff check .
.\.venv\Scripts\python.exe -m mypy app
node --test tests\frontend\*.test.mjs
node --check app\static\js\app.js
.\.venv\Scripts\python.exe scripts\analyze_reference.py
```

Тесты включают реальный ONNX-инференс на пустом и репрезентативном изображении,
OCR `007`, краткий разрыв трека, быстрый переход через горизонтальную и вертикальную
линии, интерполяцию времени, 50 кадров дрожания у линии, смену tracker ID, повторное
появление участника, одновременный проезд двух мотоциклов, конфликтующие чтения OCR,
независимый учёт `777`/`123`/`007`, API, миграции, экспорт и ошибку недоступной
камеры.

### Измеренный прогон загруженного видео

Production API версии 0.9.0 обработал пользовательский H.264-файл 848×464,
30 FPS, 137,067 с за 117,74 с на Intel Core i5-12500H и RTX 3070 Ti Laptop.
Первый проход занял 14,43 с и сформировал 12 окон/3136 кадров; второй обработал
3095 кадров на исходных 30 FPS. Это примерно 1,16× от длительности файла и в
2,55 раза быстрее сообщённого пятиминутного прогона. Создано семь корректных строк (`123`, `113`,
`004`, `15`, `222`, `044`, `72`) без посторонних номеров: precision 7/7,
recall 7/19 по читаемой пользовательской разметке. Excel-файл и отчёт проверки
созданы локально и не входят в Git.

Это измерение не подтверждает обработку трёх часов за десять минут. Точное
сопровождение каждого исходного кадра в плотной гонке и OCR маленьких размытых
табличек на данном оборудовании пока существенно медленнее такой цели. На длинной
записи с большими пустыми промежутками двухпроходный фильтр экономит больше, но
время всегда зависит от числа найденных окон и спорных OCR-событий.

## Хранение и журналы

- основная БД: `data/moto_laps.db`;
- резервные копии: `data/backups/`;
- выгрузки Excel: `data/exports/`;
- структурированный журнал: `data/logs/moto_laps.jsonl`;
- stdout/stderr фонового сервера: `data/logs/server.*.log`.
- загруженные записи и метаданные: `data/uploads/`.
- полные записи живых гонок: `data/recordings/`.

SQLite работает в WAL-режиме, имеет внешние ключи, уникальный ключ идемпотентности
проезда и уникальный номер круга участника в пределах гонки. API слушает только
loopback, ограничивает размер запроса, проверяет Origin для изменяющих запросов и
не публикует камеру в локальную сеть.

## Ограничения

- 30 FPS означает один кадр примерно каждые 33 мс, а не задержку 2–3 мс.
- Резкость быстро движущегося номера определяет выдержка камеры, а не только FPS.
- Универсальная COCO-модель обнаруживает мотоцикл, но не обучена специально на
  стартовых табличках конкретной трассы.
- Автотест не заменяет полевую калибровку с настоящим мотоциклом на 20/30/40/60
  км/ч, реальным светом и выбранной высотой камеры.
- При боковой съёмке обычной веб-камерой 30 FPS гарантировать чтение на 40–60 км/ч
  нельзя. Фронтальный верхний ракурс даёт системе больше кадров до линии.

Подробное устройство модулей описано в `docs/ARCHITECTURE.md`.
