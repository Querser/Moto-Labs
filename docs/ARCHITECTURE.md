# Архитектура Moto Laps

## Двухпроходное видео и fast-passage OCR (0.9.0)

Для загруженного файла `UploadedVideoRuntime` сначала последовательно декодирует
видео, примерно на 3 FPS проверяет движение MOG2 и запускает тот же YOLOX-Tiny.
Положительные детекции объединяются в интервалы с запасом 1,0 с до и 1,4 с после
кадра. Второй проход декодирует файл повторно, пропускает пустые промежутки и
передаёт в YOLOX/ByteTrack исходные 30 FPS только внутри найденных окон. Это
оказалось надёжнее для маленького быстрого щитка, чем 15 FPS с экстраполяцией,
при этом пустые части файла полностью исключены из тяжёлого прохода.
На границе двух независимых найденных окон сбрасывается только краткоживущее
состояние ByteTrack/OCR: пропущенный пустой интервал иначе не состарил бы старый
трек и мог бы смешать кадры двух физических мотоциклов. Записи и часы гонки при
этом не сбрасываются.
Так длинная запись не тратит OCR/ByteTrack на часы пустой трассы, но два
мотоцикла с разницей 50 мс остаются независимыми треками, а время пересечения
интерполируется по исходным capture timestamp. Первый проход не создаёт круги и
не классифицирует объект по одному лишь движению.

`MotorcycleVisionPipeline` хранит для каждого активного логического трека не более
28 краткоживущих областей-кандидатов. Короткие пространственно однозначные
ByteTrack-фрагменты склеиваются, но два близких неоднозначных мотоцикла остаются
разными. Набор сбалансирован между локализованными
светлыми табличками, общим фронтальным crop и перекрывающимися anchors, поэтому
яркий ложный контур не вытесняет единственный читаемый широкий кадр. Светлая
табличка сохраняется вместе с полями, чтобы крайняя цифра не обрезалась contour
warp; отдельные edge-кандидаты по-прежнему выправляют перспективу. При
пересечении лучшие области повторно распознаются двумя локальными OCR-сессиями;
решение требует согласованных кадров либо очень ограниченного исключительно
уверенного чтения с защитой от обрезанного префикса. Разметка
тестового видео никогда не передаётся в этот код.

В production-обработке файла исчезновение произвольного tracker ID не запускает
дорогой OCR: распознавание выполняется только для геометрического пересечения.
Диагностический evaluator может отдельно включить exit recovery, но этот путь не
пишет круги. Неразрешённый первый OCR-проход больше не уничтожает накопленный
burst: последующие кадры дополняют его. Частичные чтения разных кадров могут быть
собраны по временному перекрытию (`43` + `35` → `435`). Авторегрессионный VLM-worker
удалён из production-пути: он не улучшил контрольный ролик и мог непредсказуемо
задержать обработку.

При боковом движении горизонтальная траектория ограничивает OCR ведущими 74%
bbox. Это не даёт позднему заднему номеру заменить уже подтверждённый передний.
Для фронтального подхода ширина не ограничивается. Подтверждённая идентичность
замораживается на срок жизни физического трека. Сглаживание bbox применяется
только к SVG-оверлею; геометрия пересечения использует исходные координаты.

`RaceRecordingService` подписан на исходные кадры камеры и пишет MP4 (`mp4v`) в
отдельном потоке. Ограниченная очередь отбрасывает старейший кадр только при
аварийном отставании диска и считает такие случаи. Файл закрывается при
завершении гонки/приложения, а API выдаёт только имена с префиксом конкретной
гонки из `data/recordings/`, не принимая произвольные пути.

Разметка пользовательского MP4 лежит в
`data/reference/video_2026-07-19_13-50-08/ground_truth.csv`. Универсальный
`scripts/evaluate_labeled_video.py` сначала выполняет полный CV-пайплайн, а
ожидаемые номера читает только после инференса для расчёта precision/recall.
Отладочный `recover_pending_identities()` не создаёт кругов и используется лишь
для оценки OCR в конце размеченного временного окна; запись в БД возможна только
через реальное событие `LineCrossingDetector`.

## Границы системы

Приложение — один локальный процесс FastAPI. Он владеет камерой, выполняет CV,
изменяет SQLite и отдаёт интерфейс на `127.0.0.1`. Браузер не открывает камеру через
`getUserMedia`, поэтому нет конфликта двух владельцев устройства. Внешних сервисов
на критическом пути гонки нет.

## Поток данных

1. `app.camera` связывает имя и индекс камеры через backend-native enumeration
   (`cv2-enumerate-cameras`); в Windows GoPro Webcam открывается через DirectShow
   в исходном формате виртуального драйвера без принудительного MJPEG. Также слой читает
   безопасно загруженный файл через общий `Frame` с исходной временной меткой.
   Диспетчер передаёт исходный кадр CV и записи, а для браузера кодирует
   отдельную копию не крупнее 1280×720. Long-poll API ждёт следующий sequence и всегда
   возвращает новейший JPEG, поэтому медленная вкладка не накапливает видеозадержку.
2. Для живой камеры `VisionRuntime` отбрасывает устаревшие кадры. Для файла
   `UploadedVideoRuntime` сначала находит широкие окна MOG2 + YOLOX, затем
   повторно декодирует файл и запускает семантический трекинг на исходных 30 FPS
   только внутри найденных окон. Временная шкала всегда берётся из позиции
   исходного кадра и не зависит от скорости CPU.
3. MOG2 создаёт дешёвый сигнал движения. YOLOX-Tiny подтверждает класс
   `motorcycle`; периодический полный проход не даёт потерять неподвижный объект.
4. `SupervisionByteTrack` присваивает локальный ID и хранит временную траекторию.
5. `FrontNumberBoardRegionExtractor` ищет контрастные прямоугольные и светлые
   малонасыщенные таблички в
   верхней/передней части bbox и выправляет перспективу; резервный crop не выходит
   за bbox мотоцикла.
6. `RapidOcrDigitEngine` возвращает только цифровые строки. `OcrAggregator`
   объединяет чтения ограниченного временного окна и не преобразует строку в int.
7. `LineCrossingDetector` выбирает ведущую грань bbox по направлению траектории,
   проверяет смену стороны конечного отрезка с гистерезисом и интерполирует
   timestamp пересечения между кадрами.
8. `ParticipantPassageGuard` блокирует повтор одного физического проезда даже при
   смене tracker ID. Разблокировка происходит после выхода bbox из полосы линии
   либо после достаточного исчезновения объекта.
9. Стабильный `PassageCandidate` поступает в доменный `LapTimingService`.
   Идемпотентность и ограничения SQLite дают второй уровень защиты от дублей.
10. `LiveEventBroker` уведомляет браузер; таблица при этом всегда строится из БД,
    поэтому временная потеря WebSocket не удаляет запись.

## Время

Для камеры используется `time.monotonic_ns()`. Для загруженного видео позиция
кадра переводится в ту же монотонную шкалу относительно старта гонки. UTC сохраняется для аудита и
экспорта, но перевод системных часов не меняет длительность круга. Если объект
прошёл линию между кадрами `t0` и `t1`, время вычисляется линейной интерполяцией по
подписанному расстоянию до линии. OCR может разрешиться позже (ограниченно), однако
событие сохраняет исходный timestamp пересечения.

## Координаты линии

Два конца хранятся в диапазоне `[0, 1]`: `(x1, y1)`, `(x2, y2)`. Перед геометрией
они масштабируются к исходному кадру. Отрезок может быть горизонтальным,
вертикальным или наклонным. Проверяется именно пересечение конечного отрезка с
небольшим запасом, а не бесконечной прямой.

Frontend хранит отдельно сохранённые и локально редактируемые координаты. Пока
активен Pointer Event, имеется несохранённая правка или запрос сохранения,
периодический ответ API не может перезаписать линию. Перенос всего отрезка применяет
один общий ограниченный delta к обоим концам, сохраняя длину и угол.

## Состояние гонки и БД

`Race`, `LapRecord`, `ParticipantResult` и `CameraSetting` разделены. Новая гонка
получает новый `race_id`; круги предыдущих гонок не смешиваются с активной таблицей.
Суммарное время — сумма пересчитанных `lap_time_ns`. Паузы хранятся отдельно и
исключаются. Исправление/удаление круга атомарно перенумеровывает последующие круги.

Alembic выполняет миграции. SQLite настроен на WAL, foreign keys, busy timeout и
периодические резервные копии. Уникальны `(race_id, idempotency_key)` и
`(race_id, racing_number, lap_number)`.

## Расширение

CV-компоненты зависят от протоколов `ObjectDetector`, `ObjectTracker`,
`NumberRegionExtractor`, `OcrEngine`, `NumberVerifier`. Быстрый RapidOCR,
PP-OCRv5 mobile detector и PP-OCRv6 medium скрыты за единым `OcrEngine`;
Florence-2 работает только как консервативный tie-breaker. Для замены YOLOX,
ByteTrack или OCR
достаточно реализовать соответствующий интерфейс, не меняя доменную логику и API.
Демо-источник использует тот же pipeline contract, но передаёт эталонные metadata,
что делает интеграционные тесты воспроизводимыми без камеры.

## Безопасность и отказоустойчивость

- сервер привязан только к loopback;
- изменяющие HTTP-запросы проверяют локальный Origin;
- обычное тело запроса ограничено 1 МБ, видео — настраиваемым пределом 512 МБ;
- загрузка принимает только MP4/MOV/AVI/MKV, генерирует имя и проверяет первый кадр;
- ошибки камеры и модели превращаются в структурированное состояние, а не валят
  HTTP-процесс;
- камера и worker закрываются в lifespan/shutdown;
- одиночный экземпляр предотвращает конкуренцию за камеру и SQLite;
- логи ротируются и не содержат видеокадры;
- ORM использует параметризованный SQL.

## Монтаж камеры

Целевой вариант — фронтальный верхний ракурс: около 2 м над поверхностью,
наклон 45–60°, оптическая ось направлена в зону подхода и линию. Линия должна быть
видна, но номер — оставаться достаточно крупным до пересечения. Точные высота,
наклон, выдержка и границы линии калибруются на месте по реальным заездам.
